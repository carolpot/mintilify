Generate a new account-level webhook secret. This is an upsert operation — it creates a new secret if none exists, or replaces the existing one. The new secret is used to sign all subsequent webhook deliveries using HMAC-SHA256.

**Endpoint:** `POST https://api.lusha.com/api/account/secret/regenerate`

<Warning>
  The new secret is returned **only once** in the response. Store it immediately in your secrets manager — you cannot retrieve it again. If you lose it, you will need to regenerate again.
</Warning>

<Note>
  An account secret must exist before Lusha can deliver webhooks. Call this endpoint as your first step when setting up webhooks for the first time.
</Note>

## Behavior

| Scenario | Result |
|----------|--------|
| No secret exists | A new secret is created |
| Secret already exists | The existing secret is replaced; the old secret is immediately invalidated |

When you regenerate the secret:

- The new secret **affects all subscriptions** in your account
- The old secret is **immediately invalidated** — any in-flight delivery verification using the old secret will fail
- The new secret is **shown only once** in the response body

## Example request

```bash
curl --request POST \
  --url https://api.lusha.com/api/account/secret/regenerate \
  --header 'api_key: YOUR_API_KEY'
```

## Example response

```json
{
  "secret": "whsec_newXyz789abc..."
}
```

## Response fields

<ResponseField name="secret" type="string">
  Your new account webhook secret. Store this value immediately and securely. It will not be shown again.
</ResponseField>

## After rotating the secret

<Steps>
  <Step title="Store the secret immediately">
    Save the returned `secret` value to your secrets manager (for example, AWS Secrets Manager, HashiCorp Vault, or a secure environment variable) before closing the response. It cannot be retrieved again.
  </Step>
  <Step title="Update your endpoint handler">
    Replace the old secret in your webhook verification logic with the new value. Any requests still signed with the old secret will fail verification from this point forward.
  </Step>
  <Step title="Verify with a test delivery">
    Send a test delivery using `POST /api/subscriptions/{id}/test` to confirm that your updated endpoint correctly verifies the new signature.
  </Step>
</Steps>
